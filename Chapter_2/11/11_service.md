# [과제 2-11] ROS2 서비스(Service) 서버-클라이언트 구현 및 비동기 처리

---

## 1. ROS2 서비스(Service)와 서버-클라이언트 아키텍처의 이해

ROS2의 통신 방식은 크게 세 가지(Topic, Service, Action)로 나뉩니다. 이전까지 다루었던 **토픽(Topic)**이 데이터를 끊임없이 일방적으로 방송(Broadcast)하는 연속적인 스트림 방식이라면, **서비스(Service)**는 필요할 때만 요청하고 응답을 받는 **동기/비동기식 양방향 통신 방식**입니다.

### 1.1 서버-클라이언트(Server-Client) 아키텍처란?

- **클라이언트(Client):** 특정한 연산이나 동작을 시스템에 **요청(Request)**하는 주체입니다. 웹 브라우저에서 주소창에 URL을 입력하여 웹페이지를 달라고 요청하는 것과 같은 원리입니다.
- **서버(Server):** 클라이언트의 요청을 받아 대기하고 있다가, 요청이 들어오면 정의된 연산을 수행한 후 그 결과값을 **응답(Response)**으로 돌려주는 주체입니다.

### 1.2 ROS2 서비스의 특징

- 1:1 또는 N:1 통신이 주를 이룹니다. 하나의 서비스 서버에 여러 클라이언트가 요청을 보낼 수는 있지만, 요청 하나당 반드시 하나의 응답이 쌍(Pair)으로 이루어집니다.
- 로봇의 센서 데이터처럼 연속적인 흐름이 아니라, "특정 노드를 켜라/꺼라", "로봇의 현재 상태를 한 번 점검해라", "복잡한 연산 결과를 내놓아라" 등 **일회성이고 즉각적인 제어 명령**을 내릴 때 주로 사용됩니다.

## 2. 명령행(CLI)을 활용한 ROS2 서비스 점검 및 호출

ROS2는 터미널 명령어를 통해 현재 시스템에 활성화된 서비스의 정보를 캐내고 직접 호출(Call)할 수 있는 강력한 CLI 툴을 제공합니다. `demo_nodes_cpp` 패키지의 `add_two_ints_server` 노드를 통해 이를 검증할 수 있습니다.

### 2.1 서비스 목록 및 인터페이스 구조 확인

1. **목록 확인 (`ros2 service list`):** 현재 네트워크상에 개설된 모든 서비스의 이름을 출력합니다. (예: `/add_two_ints`)
2. **서비스 타입 확인 (`ros2 service type /add_two_ints`):** 이 서비스가 어떤 데이터 규격(인터페이스)을 사용하는지 확인합니다. 확인 결과 `example_interfaces/srv/AddTwoInts` 규격을 사용함을 알 수 있습니다.
3. **인터페이스 구조 확인 (`ros2 interface show example_interfaces/srv/AddTwoInts`):**
   (요청 데이터) `int64 a`
   (요청 데이터) `int64 b`
   --- (구분선)
   (응답 데이터) `int64 sum`

   서비스 인터페이스(`.srv`)는 토픽(`.msg`)과 달리 중간에 `---` 구분선이 존재합니다. 윗부분(`a, b`)은 클라이언트가 서버로 보내는 **요청(Request)** 데이터이며, 아랫부분(`sum`)은 서버가 계산을 마치고 돌려주는 **응답(Response)** 데이터의 규격입니다.

### 2.2 명령행으로 서비스 직접 호출하기

서비스의 이름, 타입, 그리고 요청할 인자(Argument) 값을 JSON 형태의 딕셔너리로 묶어 터미널에서 직접 서버에 전달할 수 있습니다.

> `ros2 service call /add_two_ints example_interfaces/srv/AddTwoInts "{a: 10, b: 20}"`

이 명령을 내리면 클라이언트가 서버로 `10`과 `20`을 보내고, 서버가 두 수를 더해 `30`이라는 결과를 응답(`Response`)으로 터미널에 반환하는 과정을 명확히 관찰할 수 있습니다.

## 3. 파이썬을 이용한 서비스 서버 및 클라이언트 구현 원리

### 3.1 서비스 서버(Service Server) 구현

다른 노드(또는 사용자)로부터 오는 요청을 처리하기 위해 `create_service()` 메서드를 사용합니다.

- **사용법:** `self.create_service(서비스_타입, '서비스_이름', 콜백_함수)`
- **작동 원리:** 누군가 이 서비스 이름으로 요청을 보내면 지정된 콜백 함수가 실행됩니다. 이 콜백 함수는 항상 `request`와 `response` 두 개의 매개변수를 받으며, 로직 처리가 끝난 후 반드시 `response` 객체를 반환(`return`)해야 합니다.

### 3.2 서비스 클라이언트(Service Client) 구현 및 비동기 처리

다른 노드가 제공하는 서비스를 내 노드 안에서 호출하려면 `create_client()`를 사용합니다.

- **사용법:** `self.create_client(서비스_타입, '서비스_이름')`
- **비동기 처리(Asynchronous Processing)의 필수성:** 파이썬에서 서비스 클라이언트가 서버에 요청을 보낼 때 동기식(Synchronous)으로 응답을 기다리면 시스템이 멈추는 **데드락(Deadlock, 교착상태)**에 빠집니다. ROS2의 통신 콜백을 처리하는 스레드는 기본적으로 하나인데, 응답을 기다리느라 스레드를 멈춰버리면 그 스레드가 처리해야 할 '응답 수신 작업' 자체도 막혀버리기 때문입니다.
- **해결책 (`call_async`):** 따라서 클라이언트는 `call_async(request)` 함수를 통해 비동기적으로 요청을 던집니다. 이는 "요청만 일단 던져놓고 다른 작업을 계속 진행하겠다"는 의미입니다. 응답을 받을 미래의 결과값은 `Future` 객체에 담기며, 응답이 도착했을 때 실행될 사후 처리 함수는 `add_done_callback()`으로 등록해 줍니다.

## 4. 프로그램의 정상적이고 안전한 종료(Graceful Shutdown) 매커니즘

ROS2 노드가 무한 루프(`rclpy.spin()`)를 돌고 있을 때, 강제 종료를 하면 메모리 누수가 발생하거나 예외 처리 에러(Traceback)가 터미널을 더럽히게 됩니다. 이를 우아하게 처리하는 방법은 파이썬의 **예외 처리 기법(Exception Handling)**을 사용하는 것입니다.

1. **사용자 정의 예외 발생:** 콜백 함수 내부에서 특정 조건(예: `/quit` 서비스 호출 완료)이 달성되면 인위적으로 `raise Exception("Turtle killed")`를 발생시켜 실행 흐름을 끊습니다.
2. **Main 함수에서의 예외 포획 (try-except):** 메인 스레드인 `try: rclpy.spin(node)` 블록이 이 예외를 맞고 깨어나면, 프로그램이 강제로 뻗지 않고 `except` 블록으로 넘어가 깔끔하게 `[INFO]` 로그를 출력합니다.
3. **자원 정리 (finally):** 예외 발생 여부와 상관없이 마지막에 `finally:` 구문을 타고 내려가 `rclpy.ok()`를 체크한 뒤 `rclpy.shutdown()`을 호출하여 안전하게 ROS2 통신망 접속을 해제합니다.

## 5. [부록] 평가 가이드라인 기반 심층 Q&A 검증

### Q1. 서버 - 클라이언트 아키텍처에 대해서 설명해보자.

- **답변:** 시스템의 자원이나 특정 기능을 제공하는 쪽을 **서버(Server)**, 그 기능을 필요로 할 때 요청하는 쪽을 **클라이언트(Client)**로 나누어 설계하는 구조입니다. 클라이언트가 데이터를 포함해 Request를 던지면, 서버가 이를 수신하여 내부적인 로직 처리를 끝마친 후 그 결과인 Response를 클라이언트에게 반환하는 쌍방향 상호작용 아키텍처입니다.

### Q2. ROS2의 서비스(Service)에 대해서 설명해보자.

- **답변:** 토픽처럼 연속적이고 일방적으로 데이터를 뿌리는 것이 아니라, 특정 이벤트나 명령을 수행해야 할 때만 외부 노드에서 **원격 절차 호출(RPC, Remote Procedure Call)**을 할 수 있도록 만들어진 통신 규격입니다. 일회성 제어 명령을 내리는 데 주로 사용됩니다.

### Q3. `turtlesim`의 `/kill` 서비스는 어떤 작동을 하게 되는가?

- **답변:** 클라이언트가 지우고자 하는 특정 거북이의 이름(예: `name="turtle1"`)을 담아 요청을 보내면, 시뮬레이터 화면에서 해당 거북이 객체를 완전히 삭제하고 빈 화면으로 만들어 버리는 기능을 수행합니다.

### Q4. 명령행으로 ROS2 서비스를 호출할 때 서비스에 필요한 인자를 전달하는 방법을 설명해보자.

- **답변:** 먼저 `ros2 interface show` 명령으로 해당 서비스가 요구하는 인자의 구조(이름과 자료형)를 파악합니다. 그 후 터미널에 `ros2 service call [서비스_이름] [서비스_타입] "{인자명1: 값1, 인자명2: 값2}"` 형태로, JSON 딕셔너리 포맷의 문자열로 인자 값을 매핑하여 넘겨줍니다.

### Q5. ROS2 파이썬 프로그램이 호출된 서비스를 처리하도록 구현하는 방법을 설명해보자.

- **답변:** 노드 클래스 내부에서 `self.create_service(타입, 이름, 콜백_함수)`를 호출하여 서비스 서버를 생성합니다. 클라이언트의 요청이 들어올 때마다 매핑된 콜백 함수가 실행되며, 이 함수는 인자로 들어온 `request` 데이터를 이용해 작업을 수행하고, 최종적으로 연산 결과가 담긴 `response` 객체를 리턴(반환)하여 클라이언트 측에 돌려줍니다.

### Q6. ROS2 파이썬 프로그램이 다른 노드의 서비스를 호출하는 방법을 설명해보자.

- **답변:** `self.create_client(타입, 이름)`을 이용해 클라이언트 객체를 생성합니다. 먼저 `wait_for_service()` 함수로 대상 서버가 켜져 있는지 확인한 뒤, 인자들을 채운 `Request` 객체를 만들어 `call_async(request)` 함수를 통해 비동기적으로 서버에 전송합니다.

### Q7. 파이썬 프로그램이 서비스를 호출한 후 호출 결과를 대기하지 않고 제어 노드가 계속 동작하게 만드는 방법은 무엇인가?

- **답변:** 동기식 호출을 쓰면 응답이 올 때까지 노드가 멈추기 때문에, 비동기 호출 메서드인 **`call_async()`**를 사용합니다. 호출을 던져놓은 뒤 노드가 본연의 일을 계속 수행하게 만들고, 서버의 응답이 도착했을 때만 별도로 실행될 사후 콜백 함수를 `future.add_done_callback()`을 이용해 예약해 두는 방식으로 교착상태(Deadlock)를 방지했습니다.

### Q8. `/quit` 서비스를 호출했을 때 실행 중인 노드에 어떤 방식으로 서비스가 전달되는지를 설명해보자.

- **답변:** 사용자가 명령행에서 `/quit` 서비스를 요청하면, 이 요청은 통신망을 타고 `turtle_move_control` 노드 내부에 구현된 `/quit` **서비스 서버**로 전달됩니다. 이 요청을 받은 `turtle_move_control`은 내부 로직에 따라 이번에는 자신이 **클라이언트**가 되어 `turtlesim_node`가 열어둔 `/kill` **서비스 서버**에 거북이를 죽이라는 요청을 연쇄적으로 보내게 됩니다.

### Q9. `/quit` 서비스를 처리할 때 `turtle_move_control` 노드는 어떤 방식으로 종료시켰는가?

- **답변:** 파이썬의 예외(Exception) 전파 방식을 사용했습니다. `/kill` 응답 콜백 함수 내부에서 `raise Exception("Turtle killed")`를 발생시킵니다. 이 예외는 메인 스레드로 전파되고, 메인 함수의 `try-except` 블록이 이를 포착하여 에러 메세지가 아닌 깔끔한 `[INFO]` 로그를 출력한 뒤 `rclpy.shutdown()`으로 노드를 우아하게 종료시킵니다.
