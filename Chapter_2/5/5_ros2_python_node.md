# [과제 2-5] 가상 로봇의 구성 요소를 생성한다 — ROS2 파이썬 노드 구현 및 로깅

---

## 1. ROS2 파이썬 노드(Node) 객체 생성 및 기본 구조

ROS2 프로그램은 로봇의 각 구성 요소를 '노드(Node)'로 정의하고 이들 간의 통신을 통해 구동된다. 파이썬으로 노드를 작성할 때는 객체지향 프로그래밍(OOP) 방식을 채택하여, ROS2가 제공하는 기본 노드 클래스를 상속받아 구현하는 것이 표준이다.

### 1.1 `Node` 클래스 상속

모든 사용자 정의 노드는 `rclpy.node` 모듈의 `Node` 클래스를 상속(`class LoggingNode(Node):`)받아 작성된다. 이를 통해 퍼블리셔, 서브스크라이버, 로깅, 타이머 등 ROS2의 핵심 시스템 자원을 객체 내부 메서드로 손쉽게 활용할 수 있다.

### 1.2 생명주기 제어 핵심 함수 (`rclpy`)

파이썬 노드가 실행되고 종료되기까지의 라이프사이클을 관리하는 3대 핵심 함수이다.

- **`rclpy.init()`**: ROS2 통신 네트워크(DDS)와 인터페이스를 초기화한다. 노드 생성 전 반드시 가장 먼저 호출되어야 한다.
- **`rclpy.spin(node)`**: 노드를 멈추지 않고 계속 대기(루프) 상태로 유지시키며, 타이머나 콜백 함수가 발생할 때 이를 시스템이 처리할 수 있도록 제어권을 넘긴다.
- **`rclpy.shutdown()`**: 프로그램이 종료될 때 통신망을 끊고 할당된 ROS2 자원을 안전하게 시스템에 반환한다.

---

## 2. ROS2의 로깅(Logging) 시스템

일반적인 파이썬 개발에서는 `print()` 함수를 사용해 결과를 확인하지만, 모니터가 없는 로봇 하드웨어 환경에서는 로그(Log) 파일 형태로 기록을 남기는 중앙 집중식 로깅 시스템을 사용해야 한다.

- **로그 기록 방법:** 상속받은 `Node` 객체 내부에서 `self.get_logger().info("메시지")` 형태로 호출한다.
- **로그 확인 방법:** 기본적으로 노드가 실행 중인 터미널의 표준 출력(stdout)으로 실시간 표시되며, 백그라운드에서는 `~/.ros/log/` 디렉토리 하위에 시스템 로그 파일로 누적 저장되어 사후 분석이 가능하다.

## 3. 제어 노드 파이썬 코드 구현 (`logging.py`)

`~/ros2_ws/src/my_robot_controller/my_robot_controller/` 경로에 `logging.py` 파일을 생성하고 아래 코드를 작성하였다.

```python
#!/usr/bin/env python3
import rclpy
from rclpy.node import Node

class LoggingNode(Node):
    def __init__(self):
        # 상위 클래스의 생성자를 호출하며 ROS2 네트워크 상의 노드 이름을 'logging_node'로 명명
        super().__init__('logging_node')
        # 정보(Info) 수준의 로그를 사용해 터미널에 메시지 출력
        self.get_logger().info("logging_node")

def main(args=None):
    rclpy.init(args=args)
    node = LoggingNode() # 노드 객체 생성
    rclpy.spin(node)     # 프로그램이 종료되지 않도록 무한 대기(실행 상태 유지)
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

## 4. `setup.py` 파일 수정 및 진입점(Entry Point) 등록

작성한 파이썬 소스 코드(`logging.py`)를 터미널에서 `ros2 run` 명령어로 실행할 수 있도록 빌드 시스템에 등록하는 과정이다. `setup.py` 파일 내부의 `entry_points` 영역을 수정해야 한다.

```python
# setup.py 파일의 일부 발췌
    entry_points={
        'console_scripts': [
            'logging_node = my_robot_controller.logging:main'
        ],
    },
```

- **구문 분석:** `'실행할_명령어_이름 = 패키지_이름.파이썬_파일_이름:실행할_메인함수'`
- 즉, 사용자가 터미널에 `logging_node`라고 치면, 빌드 시스템이 `my_robot_controller` 폴더 안의 `logging.py` 파일을 찾아 그 안의 `main()` 함수를 구동하라는 연결고리(매핑) 역할을 한다.

## 5. 빌드 시스템 적용 및 실행 (`--symlink-install`)

코드를 작성하고 `setup.py` 수정을 마쳤다면 변경 사항을 시스템에 반영해야 한다.

### 5.1 `--symlink-install` 옵션의 역할

파이썬 언어의 스크립트 특성을 살린 개발용 강력한 옵션이다.
기본 `colcon build`는 소스 파일을 `install` 폴더로 물리적으로 '복사'한다. 따라서 코드를 한 줄만 수정해도 매번 빌드를 다시 해야 한다. 하지만 `--symlink-install` 옵션을 주면 파일을 복사하는 대신 원본 파일의 **'바로가기(심볼릭 링크)'**를 만든다. 덕분에 **파이썬 코드를 수정하고 저장만 하면, 별도의 재빌드 과정 없이 다음 실행 시 수정 사항이 즉각 반영**되어 개발 생산성이 극대화된다.

### 5.2 시스템 적용 및 노드 실행 명령어

```bash
# 1. 워크스페이스로 이동하여 심볼릭 링크 방식으로 빌드
$cd ~/ros2_ws$ colcon build --symlink-install

# 2. 빌드된 설정 파일(오버레이)을 현재 터미널 환경 변수에 로드(적용)
$ source install/setup.bash

# 3. 작성한 노드 실행
$ ros2 run my_robot_controller logging_node
[INFO] [1733639734.465958980] [logging_node]: logging_node
```

## 6. [부록] 평가 가이드 기반 이해도 검증 문답

### Q1. ROS2의 파이썬 제어 노드를 구현하는 표준 절차를 `rclpy` 함수를 활용해 설명해보자.

- **답변:** 파이썬 환경에서 노드를 구동하려면 먼저 `rclpy.init()`으로 통신망과 인터페이스를 초기화해야 합니다. 이후 `Node`를 상속받은 사용자 정의 노드 객체를 생성하고, `rclpy.spin()` 함수에 이 객체를 전달하여 노드가 종료되지 않고 이벤트 콜백을 처리할 수 있도록 실행 상태를 묶어둡니다. 작업이 완전히 끝나면 `rclpy.shutdown()`을 호출하여 리소스를 해제하고 종료하는 것이 표준 라이프사이클입니다.

### Q2. 구현한 프로그램에서 `rclpy.spin()` 함수를 제외시키면 제어 프로그램은 어떻게 동작하는가?

- **답변:** 프로그램이 실행된 직후 단 한 번의 로그(초기화 구문)만 출력한 뒤, 대기 상태를 유지하지 못하고 파이썬 스크립트의 끝에 도달하여 **프로그램이 즉각 종료**되어 버립니다. 로봇은 지속적으로 센서를 읽고 모터를 제어해야 하므로 `spin()` 없이는 정상적인 로봇 제어가 불가능합니다.

### Q3. "정보 수준의 로그"란 무슨 의미인가? 그리고 로그 수준에는 어떤 종류가 있는가?

- **답변:** 시스템의 정상적인 실행 흐름이나 상태(예: "초기화 완료", "노드 구동 시작")를 알리는 일반적인 안내 메시지(Info)를 의미합니다. ROS2의 로깅 시스템은 상황의 심각도에 따라 총 5가지 레벨로 나뉩니다.
  1. **Debug:** 개발 단계에서 상세한 문제 추적을 위해 사용
  2. **Info:** 시스템의 일반적인 정상 동작 상태 알림 (기본 표출 기준)
  3. **Warn(Warning):** 당장 에러는 아니지만 향후 문제가 발생할 수 있는 이상 징후
  4. **Error:** 특정 기능이나 처리가 실패했으나 전체 시스템은 돌아가는 수준의 오류
  5. **Fatal(Critical):** 시스템 전체를 마비시켜 즉각 종료해야 하는 치명적 오류

### Q4. 파이썬 프로그램의 이름 `logging.py`와 노드의 이름 `logging_node` 사이에는 어떤 관계가 있는가?

- **답변:** 두 이름 사이에는 시스템적인 **직접적인 연관성이 없습니다.** `logging.py`는 단지 소스 코드가 적혀있는 물리적인 텍스트 파일의 이름(저장 위치)일 뿐이며, `logging_node`는 `super().__init__('logging_node')` 코드를 통해 ROS2 통신망에 자신을 등록할 때 사용하는 논리적인 식별 이름(주민등록번호)입니다. 파일명이 무엇이든 상관없이 노드 이름으로 네트워크 상에서 상호작용합니다.

### Q5. 제어 노드 실행을 위해서 `setup.py` 파일을 어떻게 수정했는가?

- **답변:** 패키지를 빌드할 때 운영체제가 해당 파이썬 스크립트를 찾아 실행할 수 있도록 `entry_points` 속성의 `console_scripts` 리스트에 `'logging_node = my_robot_controller.logging:main'` 문자열을 명시적으로 추가했습니다. 이는 외부에서 `ros2 run`으로 호출하는 이름(`logging_node`)과 내부 패키지의 파일(`logging.py`) 및 진입 함수(`main`)를 서로 연결(매핑)해 주는 필수 링커 역할을 수행합니다.
